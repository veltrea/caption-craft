# 字幕翻訳の設計プロトコル

LLM に字幕翻訳をやらせるときのルール。翻訳系コードを書く・修正するときに必ず守る。

---

## 守るべき3点

### 1. バッチ翻訳禁止 — 1 リクエスト = 1 リージョン

軽量 LLM (granite-tiny, ~3B 級) は **複数原文を一度に渡すと内容が似ていると勝手に圧縮統合してしまう**。

実例: 20 件入力 → 5 件にまとめて出力。JSON Schema で配列を強制しても、各要素に複数原文を詰めてくる可能性があり完全には防げない。

→ **1 リクエスト = 1 リージョンに固定**。「圧縮の余地そのもの」を消す。

---

### 2. 前後コンテキストを渡す — TARGET だけ訳させる

字幕は文の途中で切れる (例: `well done. Why does` / `it do that?`)。
1 件単独だと意味が取れず、LLM が誤訳・命令形変換・出力崩壊を起こす。

→ user prompt を以下の 3 セクションに分け、**TARGET のみ訳させる**:

- `CONTEXT BEFORE`
- `TARGET`
- `CONTEXT AFTER`

前後 2 件ずつが妥当 (5 件分のトークン増は許容範囲)。

---

### 3. JSON Schema 強制 + source エコー検証

LLM の出力フォーマットは正規表現で拾うのではなく、`response_format: {"type":"json_schema", "strict": true}` で **構造ごと強制** する。

**スキーマ:**

```json
{ "source": "TARGETを逐語エコー", "translation": "訳文" }
```

`source` エコーがあると、LLM が context を訳してしまった/原文を歪めた/別件と混同した等の不具合を **コード側で検出して warn ログに残せる**。

---

## アンチパターン (やらない)

- **`[1] xxx` `[2] yyy` 形式の番号付き出力** → 番号書式が崩れる/改行で誤解釈/圧縮の温床
- **maxTokens を増やす方向で粘る** → 圧縮は `max_tokens` 未達でも `finish_reason: stop` で起きる、トークン不足が原因ではない
- **「軽量モデルでも頑張れば JSON Schema で守れる」と思う** → 守れない。1 件ずつしか確実な方法はない

---

## モデル相性メモ (2026-05 時点 LM Studio 0.3+ 確認)

| モデル | JSON Schema | 日本語品質 | 採否 |
|---|---|---|---|
| `granite-4-h-tiny` | OK | 低 (壊れた文字列) | × |
| `mlx-community/gemma-4-26b-a4b-it` | OK | 自然 | ✓ デフォルト候補 |
| `google/gemma-3-4b` | OK | 実用レベル | ✓ |
| `qwen/qwen3-vl-8b` | OK | 実用レベル | ✓ |
| `qwen3.5-9b-mlx` | NG (content 空応答) | — | × 使えない |

新しいモデルを試すときは、**まず LM Studio に直接 JSON Schema プロンプトを叩いて content が空でないかを最初に確認** する (アプリ経由でフル検証する前に)。

---

## 経緯

「翻訳されない箇所が出る」とユーザーから報告を受けて調べた結果、軽量 LLM が文を圧縮統合していたのが原因だった。プロトコルを上記方針で再設計したところ **5/49 → 49/49** に改善。
ユーザーから「短文だと意味ないからでしょう、前後の文を見せたら」と追加指摘を受けて文脈渡しも導入。
