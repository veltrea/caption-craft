# STT (Speech-to-Text) 設計方針

CaptionCraft の音声認識エンジン選定と運用ルール。

---

## 1. 改善対象は欧州言語

日本語の Whisper 文字起こしはほぼ聞き間違いなし。**改善不要**。

本当の課題は **英語・フランス語・ドイツ語・イタリア語・スペイン語** の精度。
STT モデルの評価・選定時、日本語対応を重視しすぎない。これら欧州言語の WER 改善を主軸に評価すること。

**根拠:** ユーザーの実使用感に基づく（「日本語だったらウィスパーは聞き間違いはほとんどない」「英語、フランス語、ドイツ語、イタリア語、スペイン語あたりが怪しい」）。

---

## 2. モデルは選択肢を増やしてユーザーに選ばせる

「1つの最強モデル」を押し付けない。**用途別・言語別の選択肢を並べてユーザーが選ぶ** 方針。

- 英語だけ得意なモデルでも、得意言語を明示すれば選択肢として有効
- モデル評価時に「日本語非対応だから却下」のような一律フィルタをかけない
- 各モデルの **得意言語・サイズ・精度** を明示した上で UI のモデル選択肢に含める

---

## 3. 外部ランタイム禁止 — `.app` に完全内蔵できるものだけ

STT エンジンは `.app` バンドルに完全内蔵できるものだけ採用する。
**LM Studio・Python・pip install 等の外部ランタイムをエンドユーザーに要求してはいけない**。

**理由:** ターゲットユーザーは映像制作者であり開発者ではない。「全部便利に使えて当たり前」という層。技術的セットアップへの関心がゼロ。

**採用可能な方式:**
- SPM パッケージ (WhisperKit 方式)
- mlx-swift
- C/C++ ライブラリリンク

**開発用は別:** Python サブプロセス方式 (`SubprocessCaptionEngine`) は開発用・パワーユーザー用として残せるが、**リリース版のデフォルトにはできない**。

---

## 4. アンサンブルは並列不可 → 逐次オンデマンド方式

当初「3エンジン並列で独立列表示」を目指したが、ハードウェア制約で **並列実行は現実的に不可能** と判明 (2026-05 確認)。

### 並列が不可能な理由

1. **GPU の奪い合い**: CoreML / MLX が同じ GPU を取り合い、並列実行すると双方が遅くなる
2. **ANE のバグ・排他制約**: Neural Engine は複数モデルの同時推論に向かない (長時間ハング等)
3. **マルチスレッド非対応**: STT 推論ライブラリ自体が並列実行を想定していない
4. **コスパ**: 並列で走らせても処理時間に対するリターンが見合わない

### 現行方式 (正解)

主エンジン 1 つで全体を文字起こし → **疑わしい箇所だけ副エンジンでオンデマンド裏取り** (逐次実行)。

アンサンブル機能の改善提案で「並列実行」を提案しない。常にこの逐次オンデマンド方式を前提に設計する。
